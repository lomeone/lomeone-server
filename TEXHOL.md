# TEXHOL PRD

## 1. 문서 목적

TEXHOL은 홀덤 매장의 게임 일정, 예약, 실제 참가, 바이인/리바이 기록을 관리하는 백엔드 서비스다. 이 문서는 현재 구현된 texhol 도메인을 기준으로 제품 요구사항을 정리하고, 앞으로 API/도메인/데이터베이스를 확장할 때 기준이 되는 비즈니스 문서로 사용한다.

이 문서는 moyemap 도메인을 다루지 않는다.

## 2. 제품 개요

TEXHOL은 홀덤 매장 운영자가 매장별 게임을 등록하고, 플레이어 예약과 현장 참가 상태를 관리하며, 결제 수단별 바이인 기록을 집계할 수 있게 돕는 운영 백엔드다.

현재 서비스는 GraphQL API를 중심으로 제공되며, 핵심 객체는 매장(Store), 플레이어(Player), 게임 타입(GameType), 게임 세션(GameSession), 예약(Reservation), 게임 참가(GameEntry), 바이인 기록(BuyInRecord)이다.

## 3. 해결하려는 문제

홀덤 매장은 같은 날에도 여러 게임 타입과 여러 차수의 게임을 운영한다. 운영자는 예약자와 현장 참가자를 구분해야 하고, 예약자가 실제 참가로 전환되는 시점에 바이인 결제까지 함께 기록해야 한다. 또한 참가자는 게임 중 리바이를 할 수 있고, 일시적으로 싯아웃 상태가 될 수 있다.

TEXHOL은 이 흐름을 수기 장부나 단순 메신저 공지에 의존하지 않고, 서버에서 일관된 규칙으로 관리하는 것을 목표로 한다.

## 4. 목표

- 매장별 게임 타입과 게임 세션을 안정적으로 관리한다.
- 플레이어 닉네임을 기준으로 예약과 참가 기록을 연결한다.
- 같은 게임 세션에 같은 플레이어가 중복 예약되거나 중복 참가하지 않도록 막는다.
- 예약을 실제 참가로 전환할 때 초기 바이인 결제 기록을 자동으로 남긴다.
- 현장 참가, 리바이, 싯아웃 상태 변경을 도메인 규칙으로 관리한다.
- 결제 수단별 바이인/리바이 통계를 조회할 수 있게 한다.
- 데이터베이스 DDL과 JPA Entity의 테이블/컬럼/인덱스 구조를 일치시킨다.

## 5. 비목표

- 실시간 좌석 배치도, 테이블 밸런싱, 블라인드 타이머는 현재 범위에 포함하지 않는다.
- PG 결제 승인, 정산, 환불, 세금계산서 처리는 현재 범위에 포함하지 않는다.
- 플레이어 본인 인증, 계정 로그인, 권한 관리는 현재 texhol 도메인 핵심 범위가 아니다.
- `games` 테이블과 `Game` Entity는 레거시 호환 목적이며 신규 기능의 중심 모델로 사용하지 않는다.

## 6. 사용자

### 매장 운영자

매장을 등록하고, 매장별 게임 타입과 게임 세션을 생성한다. 예약자 목록을 확인하고, 예약자를 실제 참가자로 등록하며, 현장 참가자와 리바이 기록을 관리한다.

### 프론트엔드 서비스

texhol 사용자 화면 또는 운영 도구에서 GraphQL API를 호출한다. API는 현재 도메인 규칙에 맞게 중복 등록, 잘못된 상태 변경, 매장-게임 타입 불일치를 방어해야 한다.

### 백오피스/운영 분석 사용자

게임 세션별 참가 현황과 결제 수단별 바이인 통계를 확인한다.

## 7. 핵심 도메인 모델

### Store

매장 정보다. 매장명은 전체 서비스에서 유일해야 한다.

- 주요 필드: `id`, `name`, `location`, `address`, `imageUrl`
- 대표 규칙: 같은 이름의 매장을 중복 생성할 수 없다.
- DDL 기준: `stores.name` unique index, `location`은 도시/구 수준, `address`는 상세 주소 선택 입력

### Player

예약과 참가의 주체가 되는 플레이어다. 현재는 닉네임 기반 식별을 사용한다.

- 주요 필드: `id`, `nickname`
- 대표 규칙: 같은 닉네임의 플레이어를 중복 생성할 수 없다.
- `findOrCreatePlayer`는 닉네임이 있으면 기존 플레이어를 반환하고, 없으면 새로 생성한다.

### GameType

매장에서 운영하는 게임의 종류 또는 반복 운영 단위다. 예: 데일리, 몬스터, 메인.

- 주요 필드: `id`, `store`, `name`, `scheduleType`, `dayOfWeek`, `description`
- `scheduleType`: `DAILY`, `WEEKLY`, `MONTHLY`, `CUSTOM`
- 대표 규칙: 같은 매장 안에서 같은 게임 타입 이름을 중복 생성할 수 없다.
- 대표 규칙: `WEEKLY` 타입은 `dayOfWeek` 값이 필요하며, 값은 0부터 6 사이여야 한다.

### GameSession

실제로 운영되는 특정 게임 회차다. GameType이 게임의 종류라면 GameSession은 해당 게임의 특정 회차다.

- 주요 필드: `id`, `store`, `gameType`, `session`, `status`
- `status`: `RECRUITING`, `EARLY_CLOSE`, `CLOSED`
- 대표 규칙: 같은 매장, 같은 게임 타입, 같은 차수의 게임 세션은 하나만 존재할 수 있다.
- 대표 규칙: 게임 세션 생성 시 `gameType.store`와 요청된 `store`가 일치해야 한다.

### Reservation

플레이어가 특정 게임 세션에 참여 의사를 밝힌 예약 기록이다.

- 주요 필드: `id`, `gameSession`, `player`, `time`, `status`
- `status`: `WAITING`, `REGISTERED`, `CANCELLED`
- 대표 규칙: 같은 게임 세션에 같은 플레이어는 하나의 예약만 가질 수 있다.
- 대표 규칙: 예약은 `WAITING` 상태에서만 참가 등록으로 전환된다.
- 대표 규칙: 이미 취소된 예약을 다시 취소할 수 없다.

### GameEntry

실제 게임에 참가한 플레이어의 엔트리다. 예약을 등록 처리하거나, 현장 참가로 직접 생성할 수 있다.

- 주요 필드: `id`, `gameSession`, `player`, `status`, `buyInRecords`, `reBuyCount`
- `status`: `ALIVE`, `SIT_OUT`
- 대표 규칙: 같은 게임 세션에 같은 플레이어는 하나의 엔트리만 가질 수 있다.
- 대표 규칙: 엔트리 생성 시 초기 바이인 기록이 반드시 함께 생성된다.
- 대표 규칙: 리바이는 `ALIVE` 상태의 엔트리에만 추가할 수 있다.

### BuyInRecord

게임 참가자의 초기 바이인과 리바이 이력을 나타낸다. `GameEntry`에 종속되는 값 객체 컬렉션이다.

- 주요 필드: `type`, `paymentMethod`, `createdAt`
- `type`: `INITIAL`, `RE_BUY`
- `paymentMethod`: `CASH`, `CARD`, `POINTS`
- 대표 규칙: 엔트리 최초 생성 시 `INITIAL` 기록이 생성되고, 리바이 시 `RE_BUY` 기록이 추가된다.

## 8. 핵심 사용자 흐름

### 8.1 매장 등록

1. 운영자가 매장명, 위치, 상세 주소, 이미지 URL을 입력한다.
2. 서버는 매장명 중복 여부를 확인한다.
3. 중복이 없으면 매장을 생성한다.
4. 생성된 매장은 게임 타입 생성의 기준점이 된다.

### 8.2 게임 타입 등록

1. 운영자가 특정 매장에 게임 타입을 등록한다.
2. 서버는 매장 존재 여부와 매장 내 게임 타입명 중복 여부를 확인한다.
3. 주간 게임이면 요일 값의 유효성을 검증한다.
4. 생성된 게임 타입은 게임 세션 생성 시 사용된다.

### 8.3 게임 세션 생성

1. 운영자가 매장, 게임 타입, 차수를 지정한다.
2. 서버는 매장과 게임 타입이 같은 매장에 속하는지 검증한다.
3. 같은 매장/게임 타입/차수 조합이 이미 있으면 생성하지 않는다.
4. 세션은 기본적으로 모집 중(`RECRUITING`) 상태로 시작한다.

### 8.4 예약 생성

1. 운영자가 플레이어, 게임 세션, 예약 시간을 입력한다.
2. 서버는 같은 게임 세션에 같은 플레이어 예약이 있는지 확인한다.
3. 중복이 없으면 `WAITING` 상태의 예약을 생성한다.

### 8.5 예약자 참가 등록

1. 운영자가 예약자를 실제 참가자로 등록한다.
2. 서버는 예약 상태가 `WAITING`인지 확인한다.
3. 같은 게임 세션에 같은 플레이어의 엔트리가 이미 있는지 확인한다.
4. 엔트리를 생성하고 초기 바이인 기록을 남긴다.
5. 예약 상태를 `REGISTERED`로 변경한다.

### 8.6 현장 참가 등록

1. 운영자가 예약 없이 바로 참가한 플레이어를 게임 엔트리로 등록한다.
2. 서버는 같은 게임 세션에 같은 플레이어 엔트리가 있는지 확인한다.
3. 엔트리를 생성하고 초기 바이인 기록을 남긴다.

### 8.7 리바이 등록

1. 운영자가 엔트리와 결제 수단을 선택해 리바이를 등록한다.
2. 서버는 엔트리가 존재하고 `ALIVE` 상태인지 확인한다.
3. 리바이 기록을 추가한다.
4. `reBuyCount`는 `RE_BUY` 타입 기록의 개수로 계산된다.

### 8.8 싯아웃 토글

1. 운영자가 엔트리의 참가 상태를 변경한다.
2. 서버는 `ALIVE`와 `SIT_OUT` 상태를 서로 전환한다.
3. 싯아웃 상태의 엔트리는 리바이 등록 대상에서 제외된다.

### 8.9 결제 통계 조회

1. 운영자가 특정 게임 세션을 선택한다.
2. 서버는 해당 세션의 엔트리와 바이인 기록을 조회한다.
3. 결제 수단별 바이인/리바이 개수를 집계한다.

## 9. 비즈니스 규칙

- 매장명은 전체 서비스에서 유일해야 한다.
- 플레이어 닉네임은 전체 서비스에서 유일해야 한다.
- 게임 타입 이름은 같은 매장 안에서 유일해야 한다.
- 게임 세션은 `storeId + gameTypeId + session` 조합으로 유일해야 한다.
- 게임 타입은 자신이 속한 매장에 대해서만 세션을 만들 수 있다.
- 예약은 `gameSessionId + playerId` 조합으로 유일해야 한다.
- 엔트리는 `gameSessionId + playerId` 조합으로 유일해야 한다.
- 예약 등록은 `WAITING` 상태에서만 가능하다.
- 예약 취소는 이미 `CANCELLED` 상태인 경우 실패해야 한다.
- 예약 플레이어 변경 시 변경 대상 플레이어가 같은 세션에 이미 예약되어 있으면 실패해야 한다.
- 엔트리 플레이어 변경 시 변경 대상 플레이어가 같은 세션에 이미 참가 중이면 실패해야 한다.
- 리바이는 `ALIVE` 상태의 엔트리에만 가능하다.
- 바이인 기록은 엔트리의 하위 기록이며 독립 엔티티로 직접 생성하지 않는다.

## 10. API 요구사항

TEXHOL API는 GraphQL 스키마를 통해 제공된다.

### Query

- `store(id)`: 매장 단건 조회
- `stores`: 전체 매장 조회
- `player(id)`: 플레이어 단건 조회
- `playerByNickname(nickname)`: 닉네임 기반 플레이어 조회
- `gameType(id)`: 게임 타입 단건 조회
- `gameTypesByStore(storeId)`: 매장별 게임 타입 목록 조회
- `gameSession(id)`: 게임 세션 단건 조회
- `gameSessionsByStore(storeId, status)`: 매장별 게임 세션 목록 조회, 상태 필터 지원
- `reservationsByGameSession(gameSessionId)`: 게임 세션별 예약 목록 조회
- `reservationsByPlayer(playerId)`: 플레이어별 예약 목록 조회
- `gameEntry(id)`: 게임 엔트리 단건 조회
- `gameEntriesByGameSession(gameSessionId)`: 게임 세션별 엔트리 목록 조회

### Mutation

- `createStore`: 매장 생성
- `createPlayer`: 플레이어 생성
- `findOrCreatePlayer`: 플레이어 조회 또는 생성
- `createGameType`: 게임 타입 생성
- `createGameSession`: 게임 세션 생성
- `updateGameSessionStatus`: 게임 세션 상태 변경
- `createReservation`: 예약 생성
- `registerReservation`: 예약자를 실제 참가자로 등록
- `cancelReservation`: 예약 취소
- `changeReservationPlayer`: 예약 플레이어 변경
- `createGameEntry`: 현장 참가 또는 직접 참가 엔트리 생성
- `addReBuy`: 리바이 기록 추가
- `toggleSitOut`: 엔트리 상태 토글
- `changeGameEntryPlayer`: 엔트리 플레이어 변경

## 11. 데이터베이스 요구사항

현재 기준 DDL은 `schema/texhol_ddl.sql`을 기준으로 한다.

### 필수 테이블

- `stores`
- `players`
- `game_types`
- `game_sessions`
- `reservations`
- `game_entries`
- `buy_in_records`

### 레거시 테이블

- `games`: 과거 문자열 기반 게임 모델이다. 신규 기능에서는 `game_types`와 `game_sessions`를 사용한다.

### 필수 Unique Index

- `stores.name`
- `players.nickname`
- `game_types.store_id + name`
- `game_sessions.store_id + game_type_id + session`
- `reservations.game_session_id + player_id`
- `game_entries.game_session_id + player_id`
- `games.store_id + game_type + session` 레거시 호환

### 조회 성능을 위한 Index

- `game_sessions.store_id + status`
- `game_sessions.game_type_id`
- `reservations.player_id`
- `game_entries.player_id`
- `buy_in_records.game_entry_id`

## 12. 품질 요구사항

- 도메인 규칙은 API 계층이 아니라 domain 서비스와 Entity에서 우선 보장해야 한다.
- 중복 생성은 애플리케이션 레벨에서 먼저 검증하고, 데이터베이스 unique index로 한 번 더 방어한다.
- 상태 전이는 명시적인 메서드 또는 유스케이스 서비스를 통해 수행한다.
- 테스트는 Kotest 기반으로 작성하며, 핵심 비즈니스 규칙은 domain 테스트에서 검증한다.
- Entity와 DDL의 테이블명, 컬럼명, PK, unique index는 항상 동기화되어야 한다.
- 외래키는 운영 정책상 DDL에 명시하지 않을 수 있으나, 애플리케이션 도메인에서는 관계 무결성을 유지해야 한다.

## 13. 운영 요구사항

- 게임 세션 상태별 목록 조회가 빨라야 한다.
- 예약자 등록과 현장 참가 등록은 중복 참가를 만들지 않아야 한다.
- 결제 통계는 운영자가 현장에서 즉시 확인할 수 있을 만큼 단순하고 빠르게 계산되어야 한다.
- 이미지 URL은 현재 `directory/filename` 형태의 내부 규칙을 전제로 하며, 파일명이 난수를 포함해도 255자 이내로 관리한다.
- 위치(`location`)는 도시/구 수준 값을 저장한다. 예: `서울 관악구`, `부산 서면구`

## 14. MVP 범위

MVP에서 반드시 제공해야 하는 기능은 다음과 같다.

- 매장 생성 및 목록 조회
- 플레이어 생성, 닉네임 기반 조회, 없으면 생성
- 매장별 게임 타입 생성 및 조회
- 매장별 게임 세션 생성, 조회, 상태 변경
- 게임 세션별 예약 생성, 변경, 취소, 등록
- 게임 세션별 참가 엔트리 생성 및 조회
- 리바이 등록
- 싯아웃 상태 변경
- 결제 수단별 바이인/리바이 통계

## 15. 향후 논의 사항

- 게임 세션에 날짜 또는 운영일 개념을 명시적으로 추가할지 결정해야 한다.
- 예약 시간(`time`)을 문자열로 유지할지, 시간 타입으로 정규화할지 결정해야 한다.
- 플레이어 닉네임만으로 식별할 때 동명이인/닉네임 변경 문제를 어떻게 처리할지 정해야 한다.
- 게임 종료 이후 엔트리 변경, 리바이 추가, 예약 취소를 제한할지 상태 전이 정책이 필요하다.
- 매장 운영자 권한과 일반 사용자 권한을 분리할지 결정해야 한다.
- 결제 수단 외에 금액, 쿠폰, 포인트 차감 등 정산 정보를 저장할지 결정해야 한다.
- `games` 레거시 모델의 제거 시점과 마이그레이션 경로를 정해야 한다.
