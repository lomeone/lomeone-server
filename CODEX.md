## Project Role

`lomeone-server` is a shared backend server for two product domains:

- `texhol`: poker/game store operations, sessions, entries, buy-ins, players,
  and reservations.
- `moyemap`: venue/map discovery, reviews, ratings, regions, and source links.

The project is also intended to host future Lomeone backend domains, so keep
domain boundaries explicit when adding code.

## Current Shape

This is a Kotlin/Spring Boot multi-module Gradle project:

```text
api
├── depends on domain
└── depends on infrastructure

domain
└── depends on util

infrastructure
└── depends on domain

util
```

- `api`: Spring Boot application entry point and DGS GraphQL data fetchers.
- `domain`: JPA entities, repository interfaces, domain exceptions, and use-case
  services for `texhol`, `moyemap`, and common value objects.
- `infrastructure`: data source, AWS, and OpenFeign configuration. Repository
  implementations are currently minimal because most repositories are Spring
  Data `JpaRepository` interfaces in `domain`.
- `util`: shared utility code such as crypto converters.

## Important Source Areas

- Entry point: `api/src/main/kotlin/com/lomeone/LomeoneApiApplication.kt`
- GraphQL schemas: `api/src/main/resources/schema/`
- Generated DGS types package: `com.lomeone.generated`
- Texhol domain: `domain/src/main/kotlin/com/lomeone/texhol/`
- Moyemap domain: `domain/src/main/kotlin/com/lomeone/moyemap/`
- Shared common domain: `domain/src/main/kotlin/com/lomeone/common/`
- Infrastructure config: `infrastructure/src/main/kotlin/com/lomeone/config/`
  and `infrastructure/src/main/kotlin/com/lomeone/texhol/config/`

## Implementation Notes

- Prefer the existing use-case service style: small Spring `@Service` classes
  with command/query/result data classes near the service.
- Keep product packages separate: add texhol code under
  `com.lomeone.texhol...` and moyemap code under `com.lomeone.moyemap...`.
- Keep API mapping logic in DGS data fetchers unless a shared mapper already
  exists.
- Use Kotest for tests.
- QueryDSL Q-classes are generated by KSP during compilation and should not be
  edited by hand.
- JPA entities are opened by the Kotlin allOpen/JPA plugin; avoid unnecessary
  manual `open` declarations.
- Be careful with lazy JPA relationships in GraphQL fetchers. Several current
  data fetchers map nested entities directly, so service transaction boundaries
  and fetch strategies matter.
- The current API layer is GraphQL-first. Older REST reservation files appear to
  have been removed in the working tree.

## Current Domain Coverage

### Texhol

Implemented source areas include:

- Stores: create/list/get.
- Players: create/find-or-create/get by nickname.
- Game types and sessions: create/list/get/status updates.
- Game entries: create, re-buy, sit-out toggle, player changes, payment stats.
- Reservations: create/register/cancel/list/change player.

GraphQL data fetchers exist for stores, players, game types, game sessions, game
entries, and reservations.

### Moyemap

Implemented source areas include:

- Venues with category, embedded location, price, image, description, source URL,
  tags/images/amenities stored as JSON text for now, and average rating.
- Reviews linked to venues.
- Venue query filtering by category, region, price, rating, and rating order.
- GraphQL data fetchers for venues and reviews.

## Build And Test

Useful commands:

```bash
./gradlew build
./gradlew test
./gradlew :domain:test
./gradlew :api:test
./gradlew :infrastructure:test
./gradlew koverHtmlReport
./gradlew koverXmlReport
```

Environment:

- Java 21 is required.
- `GITHUB_ACTOR` and `GITHUB_TOKEN` are required for the GitHub Packages Maven
  registry that hosts `eunoia`.

## Known Cautions

- The working tree may contain broad in-progress changes. Do not revert or
  rewrite user changes unless explicitly asked.
- Root docs mention REST APIs as a goal, but the current codebase is oriented
  around GraphQL/DGS.
- Application name in `application.yaml` is still `texhol-server`, even though
  this repository now serves both texhol and moyemap.
- `application-prod.yaml` is currently empty.
- Coverage minimum is configured as `0`, so passing coverage verification does
  not imply meaningful coverage.
