# MOYEMAP PRD

## 1. 제품 개요

Moyemap은 전국의 소셜링 이벤트를 하나의 지도에서 탐색할 수 있게 하는 서비스다.
사용자는 소셜링, 솔로파티, 게스트하우스파티, 로테이션소개팅, 네트워킹, 혼술바 등
오프라인 만남/교류 이벤트를 지역, 날짜, 카테고리, 가격, 분위기, 모집 상태 기준으로
빠르게 찾고 원본 예약 페이지로 이동할 수 있다.

이 문서는 `moyemap.lomeone.com` 프론트엔드가 더미 데이터에서 실제 서버 데이터로
전환하기 위해 필요한 서버 요구사항을 정의한다.

## 2. 배경과 문제

현재 소셜링 정보는 각 플랫폼, 호스트, 장소, SNS, 예약 페이지에 흩어져 있다. 사용자는
원하는 지역과 날짜에 참여 가능한 모임을 찾기 위해 여러 서비스를 반복적으로 검색해야
한다. 특히 지도 기반으로 한눈에 볼 수 있는 탐색 경험이 부족하고, 같은 장소에서 열리는
다른 이벤트나 같은 카테고리의 대체 이벤트를 비교하기 어렵다.

Moyemap 서버는 여러 출처의 이벤트 정보를 표준화하고, 지도에 적합한 위치 데이터와
검색/필터링 API를 제공하며, 추후 사용자 저장, 리뷰, 클릭 추적, 추천 기능으로 확장할 수
있는 기반을 제공해야 한다.

## 3. 목표

- 전국 소셜링 이벤트를 지도와 리스트에서 빠르게 조회할 수 있는 API를 제공한다.
- 이벤트, 장소, 호스트, 외부 원본 링크를 분리해 중복 데이터를 줄이고 재사용성을 높인다.
- 지역, 카테고리, 날짜, 가격, 모집 상태, 키워드, 지도 영역 기반 검색을 지원한다.
- 프론트엔드 더미 데이터를 서버 데이터로 대체할 수 있는 최소 기능을 우선 완성한다.
- 외부 원본 페이지 이동과 클릭 추적을 통해 인기 이벤트와 전환 데이터를 쌓는다.
- 향후 데이터 수집, 관리자 검수, 사용자 저장/리뷰, 개인화 추천으로 확장 가능한 구조를 만든다.

## 4. 비목표

- Moyemap 자체 결제와 예약 완료 처리는 초기 범위에 포함하지 않는다.
- 채팅, 커뮤니티 피드, DM 기능은 초기 범위에 포함하지 않는다.
- 외부 플랫폼의 실시간 좌석/재고와 완전 동기화하는 것은 초기 범위에 포함하지 않는다.
- 지도 타일, 경로 탐색, 내비게이션 기능은 서버가 직접 제공하지 않는다.
- 모든 이벤트 정보의 법적/상업적 사용 권한 자동 판정은 서버 기능이 아니라 운영 정책으로 관리한다.

## 5. 주요 사용자

### 일반 사용자

- 가까운 소셜링 이벤트를 지도에서 찾고 싶은 사용자
- 이번 주말 또는 특정 날짜에 참여 가능한 모임을 비교하려는 사용자
- 솔로파티, 게스트하우스파티, 네트워킹, 혼술바처럼 목적이 명확한 이벤트를 찾는 사용자
- 이벤트 상세 정보를 확인하고 원본 예약 페이지로 이동하려는 사용자

### 운영자

- 외부 이벤트 정보를 등록, 수정, 숨김 처리하는 내부 운영자
- 잘못된 위치, 중복 이벤트, 종료된 이벤트를 정리하는 관리자
- 인기 카테고리, 지역, 클릭 수를 확인해 데이터 품질과 운영 우선순위를 결정하는 담당자

### 개발자/시스템

- 프론트엔드 지도/리스트 화면에 필요한 조회 API를 사용하는 클라이언트
- 외부 데이터 수집기, 배치 작업, 관리자 도구
- 관측성, 알림, 통계 파이프라인

## 6. 핵심 사용자 여정

### 지도 탐색

1. 사용자가 Moyemap에 진입한다.
2. 서버는 기본 지역 또는 현재 지도 영역에 해당하는 이벤트 마커를 반환한다.
3. 사용자가 지도를 이동하거나 확대/축소하면 서버는 지도 경계 내 이벤트를 다시 조회한다.
4. 사용자는 마커를 눌러 요약 정보를 보고 상세 화면으로 이동한다.

### 필터 탐색

1. 사용자가 카테고리, 날짜, 가격, 지역, 모집 상태를 선택한다.
2. 서버는 조건에 맞는 이벤트 목록과 지도 마커 데이터를 반환한다.
3. 사용자는 정렬 기준을 변경해 가까운 순, 임박한 순, 인기 순, 가격 낮은 순으로 비교한다.

### 상세 확인과 외부 이동

1. 사용자가 이벤트 상세를 조회한다.
2. 서버는 이벤트 설명, 장소, 일정, 가격, 태그, 이미지, 호스트, 원본 링크를 반환한다.
3. 사용자가 원본 페이지 버튼을 누르면 서버는 클릭 로그를 기록한다.
4. 클라이언트는 원본 예약/소개 페이지로 이동시킨다.

### 운영자 등록/검수

1. 운영자 또는 수집기가 이벤트 후보 데이터를 등록한다.
2. 서버는 중복 가능성, 필수 필드, 위치 좌표 유효성을 검증한다.
3. 운영자는 이벤트를 게시, 수정, 숨김, 종료 처리한다.
4. 게시된 이벤트만 일반 사용자 검색 결과에 노출된다.

## 7. MVP 범위

- 이벤트와 장소 연결

- 원본 링크 클릭 로그 기록
- 이미지 URL, 태그, 분위기, 가격 정보 제공
- 기본 관리자용 API

### MVP 이후 기능

- 사용자 계정 기반 저장/찜

- 이벤트 신고
- 외부 데이터 수집기와 자동 갱신
- 중복 이벤트 자동 병합 후보
- 인기/추천 랭킹
- 지역별 SEO 페이지용 데이터 API
- 알림 또는 구독

## 8. 도메인 정의

### Event

지도와 리스트에 노출되는 핵심 단위다. 소셜링, 솔로파티, 게스트하우스파티,
로테이션소개팅, 네트워킹, 혼술바 같은 실제 참여 가능한 모임 또는 장소형 경험을
포괄한다.

필수 속성:

- 이벤트 ID
- 제목
- 카테고리
- 요약 설명
- 상세 설명
- 대표 이미지 URL
- 시작 일시
- 종료 일시
- 가격
- 통화
- 모집 상태
- 게시 상태
- 원본 URL
- 장소 ID
- 호스트 ID
- 태그
- 생성/수정 시각

선택 속성:

- 연령대
- 성별 조건
- 최소/최대 참여 인원
- 남은 좌석 수
- 반복 이벤트 그룹 ID
- 외부 플랫폼 이름
- 외부 원본 ID
- 원본 마지막 확인 시각
- 검수 상태

### Venue

이벤트가 열리는 물리적 장소 또는 상시 운영되는 장소형 소셜링 지점이다.

필수 속성:

- 장소 ID
- 장소명
- 주소
- 위도
- 경도
- 행정 지역
- 대표 이미지 URL
- 장소 설명
- 게시 상태

선택 속성:

- 도로명 주소
- 지번 주소
- 건물/층/호수
- 전화번호
- 영업 시간
- 편의시설
- 장소 태그
- 외부 지도 장소 ID
- 원본 URL
- 평균 평점

### Host

이벤트를 운영하거나 소개하는 주체다. 개인, 브랜드, 게스트하우스, 바, 커뮤니티,
외부 플랫폼이 될 수 있다.

필수 속성:

- 호스트 ID
- 호스트명
- 유형
- 게시 상태

선택 속성:

- 소개
- 프로필 이미지 URL
- 웹사이트 URL
- SNS URL
- 연락처

### Category

사용자 탐색과 운영 분류의 기준이다.

초기 카테고리:

- `SOCIALING`: 소셜링
- `SOLO_PARTY`: 솔로파티
- `GUESTHOUSE_PARTY`: 게스트하우스파티
- `ROTATION_DATING`: 로테이션소개팅
- `NETWORKING`: 네트워킹
- `HONSOOL_BAR`: 혼술바
- `BAR`: 바/펍
- `WORKSHOP`: 클래스/워크숍
- `ETC`: 기타

카테고리는 프론트 표시명과 내부 enum을 분리한다. 카테고리 변경이 잦을 수 있으므로
초기에는 enum으로 시작하되, 운영 확장 시 별도 테이블화를 검토한다.

### SourceLink

원본 예약/소개 페이지와 추적 정보를 관리한다.

필수 속성:

- 링크 ID
- 이벤트 ID
- URL
- 원본 플랫폼
- 활성 여부

선택 속성:

- UTM 파라미터
- 마지막 수집 시각
- HTTP 상태
- 클릭 수

### ClickLog

외부 이동 이벤트를 기록한다. 개인 식별 정보는 최소화하고, 분석에 필요한 범위만 저장한다.

필수 속성:

- 로그 ID
- 이벤트 ID
- source link ID
- 클릭 시각
- referrer 또는 화면 위치

선택 속성:

- 익명 사용자 ID
- 세션 ID
- user agent
- IP 기반 지역
- 필터 조건 snapshot

## 9. 상태 정의

### EventStatus

- `DRAFT`: 등록 중이며 사용자에게 노출되지 않음
- `PUBLISHED`: 사용자에게 노출됨
- `HIDDEN`: 운영자에 의해 숨김 처리됨
- `ENDED`: 일정이 종료됨
- `CANCELED`: 취소됨

### RecruitingStatus

- `OPEN`: 모집 중
- `WAITLIST`: 대기 신청 가능
- `SOLD_OUT`: 마감
- `CLOSED`: 신청 종료
- `UNKNOWN`: 원본에서 확인 불가

### ReviewStatus

- `PENDING`: 검수 대기
- `PUBLISHED`: 노출 가능
- `HIDDEN`: 숨김
- `REJECTED`: 반려

## 10. 기능 요구사항

### 이벤트 조회

- 사용자는 게시된 이벤트 목록을 조회할 수 있다.
- 사용자는 이벤트 ID로 상세 정보를 조회할 수 있다.
- 사용자는 지도 경계값으로 해당 영역의 이벤트를 조회할 수 있다.
- 서버는 지도 마커용 경량 응답과 상세 응답을 분리한다.
- 종료, 취소, 숨김 상태의 이벤트는 기본 조회 결과에서 제외한다.
- 사용자가 `includeEnded` 같은 명시 조건을 제공하면 종료 이벤트를 포함할 수 있다.

### 검색과 필터

- 사용자는 키워드로 제목, 장소명, 지역, 태그를 검색할 수 있다.
- 사용자는 카테고리 다중 선택으로 검색할 수 있다.
- 사용자는 시작/종료 날짜 범위로 검색할 수 있다.
- 사용자는 가격 범위로 검색할 수 있다.
- 사용자는 무료 이벤트만 검색할 수 있다.
- 사용자는 모집 상태로 검색할 수 있다.
- 사용자는 지역 단위로 검색할 수 있다.
- 사용자는 현재 지도 영역 내 이벤트만 검색할 수 있다.
- 서버는 페이지네이션 또는 커서 기반 목록 조회를 제공한다.

### 정렬

- 기본 정렬은 임박한 일정 순이다.
- 사용자는 가까운 순, 인기 순, 최신 등록 순, 가격 낮은 순, 가격 높은 순, 평점 순으로 정렬할 수 있다.
- 가까운 순 정렬은 기준 좌표가 제공된 경우에만 사용한다.
- 인기 순은 클릭 수, 상세 조회 수, 저장 수를 조합해 계산할 수 있다.

### 지도 마커

- 서버는 지도에 필요한 최소 필드만 반환하는 마커 API를 제공한다.
- 동일 좌표 또는 가까운 좌표에 여러 이벤트가 있을 수 있으므로 클러스터링에 필요한 count를 제공할 수 있다.
- 지도 영역이 너무 넓은 경우 서버는 결과 수 제한, 클러스터 응답, 또는 축약 응답을 적용한다.

### 장소

- 운영자는 장소를 등록하고 수정할 수 있다.
- 장소는 여러 이벤트에 연결될 수 있다.
- 장소 좌표는 필수이며 위도/경도 범위를 검증한다.
- 주소와 좌표가 불일치하는 경우 운영 검수 상태로 표시할 수 있다.
- 같은 이름과 가까운 좌표의 장소는 중복 후보로 식별할 수 있어야 한다.

### 외부 링크와 클릭 추적

- 이벤트는 하나 이상의 원본 링크를 가질 수 있다.
- 사용자가 원본 링크를 클릭하면 서버는 클릭 로그를 기록한다.
- 클릭 로그 기록 실패가 사용자 이동을 막아서는 안 된다.
- 비활성 링크는 일반 사용자 응답에서 제외한다.
- 원본 URL은 운영자가 수정할 수 있다.

### 리뷰와 평점

- 초기 MVP에서는 선택 기능으로 둔다.
- 사용자는 이벤트 또는 장소에 리뷰를 작성할 수 있다.
- 평점은 1점에서 5점 사이만 허용한다.
- 리뷰는 검수 상태를 가질 수 있다.
- 평균 평점은 게시된 리뷰만 반영한다.

### 사용자 기능

- 비로그인 사용자는 이벤트 탐색과 외부 이동을 사용할 수 있다.
- 로그인 사용자는 이벤트 저장/찜을 사용할 수 있다.
- 로그인 사용자는 리뷰, 신고, 알림 구독을 사용할 수 있다.
- 사용자 식별은 향후 공통 인증 체계와 연동한다.

### 관리자 기능

- 운영자는 이벤트, 장소, 호스트를 등록/수정/숨김 처리할 수 있다.
- 운영자는 잘못된 위치, 종료된 이벤트, 중복 이벤트를 정리할 수 있다.
- 운영자는 원본 링크 상태와 마지막 확인 시각을 볼 수 있다.
- 운영자는 카테고리, 태그, 이미지, 가격, 모집 상태를 수정할 수 있다.
- 관리자 API는 일반 API와 권한을 분리한다.

### 데이터 수집

- 수집기는 외부 원본 데이터를 후보 이벤트로 등록할 수 있다.
- 후보 이벤트는 게시 전 검수 상태를 가질 수 있다.
- 외부 원본 ID와 플랫폼명을 저장해 중복 수집을 방지한다.
- 원본의 일정, 가격, 모집 상태가 바뀔 수 있으므로 마지막 확인 시각을 저장한다.
- 수집 실패, 원본 접근 불가, 파싱 실패는 별도 로그로 남긴다.

## 11. API 요구사항

현재 서버는 Netflix DGS GraphQL을 중심으로 구성되어 있으므로 Moyemap도 GraphQL을
우선 제공한다. 지도 이동처럼 짧고 빈번한 요청은 추후 REST 또는 전용 lightweight endpoint를

- 커서 또는 페이지

초기 구현은 MySQL + JPA로 시작할 수 있다. 지도 검색 성능이 병목이 되면
PostgreSQL/PostGIS 또는 검색 엔진 도입을 검토한다.

### 주요 테이블

- `moyemap_venues`
- `moyemap_hosts`
- `moyemap_source_links`
- `moyemap_click_logs`
- `moyemap_reviews`
- `moyemap_saved_events`
- `moyemap_event_tags`
- `moyemap_import_jobs`
- `moyemap_import_errors`

### 인덱스

- 이벤트 게시 상태 + 시작 일시
- 이벤트 카테고리 + 시작 일시
- 이벤트 모집 상태 + 시작 일시

MVP에서는 위도/경도 범위 조건으로 지도 영역 검색을 구현한다.
정확한 반경 검색, 거리 정렬, 클러스터링이 중요해지면 다음 중 하나를 선택한다.

- MySQL spatial index
- PostgreSQL/PostGIS
- OpenSearch geo query
- Redis GEO

## 13. 검증 규칙

- 이벤트 제목은 비어 있을 수 없다.
- 이벤트 시작 일시는 종료 일시보다 빠르거나 같아야 한다.
- 가격은 0 이상이어야 한다.
- 위도는 -90 이상 90 이하, 경도는 -180 이상 180 이하만 허용한다.
- 원본 URL은 유효한 HTTP/HTTPS URL이어야 한다.
- 게시 상태로 변경하려면 제목, 카테고리, 장소, 시작 일시, 원본 URL이 필요하다.
- 종료된 이벤트는 기본적으로 검색 결과에서 제외한다.
- 같은 원본 플랫폼과 외부 원본 ID의 이벤트는 중복 등록할 수 없다.
- 같은 장소, 같은 시작 일시, 유사 제목의 이벤트는 중복 후보로 표시한다.

## 14. 권한 요구사항

- 일반 조회 API는 비로그인 접근을 허용한다.
- 이벤트 저장, 리뷰, 신고, 알림은 로그인 사용자를 요구한다.
- 이벤트/장소/호스트 생성과 수정은 운영자 권한을 요구한다.
- 게시, 숨김, 삭제, 중복 병합은 관리자 권한을 요구한다.
- 클릭 추적은 비로그인에서도 가능하되 개인정보 저장을 최소화한다.

## 15. 비기능 요구사항

### 성능

- 첫 지도 로딩용 마커 API는 일반적인 도심 영역 기준 300ms 이내 응답을 목표로 한다.
- 목록 조회는 기본 20개, 최대 100개 단위로 제한한다.
- 지도 영역이 넓거나 결과가 많은 경우 서버는 결과 수를 제한하거나 클러스터 응답을 제공한다.
- 상세 조회는 이미지 목록과 리뷰를 필요 시 분리 로딩할 수 있어야 한다.

### 안정성

- 클릭 로그 저장 실패는 원본 페이지 이동을 차단하지 않는다.
- 외부 원본 수집 실패는 기존 게시 데이터를 자동 삭제하지 않는다.
- 데이터 수집과 사용자 조회 API는 장애 전파를 막기 위해 분리한다.

### 관측성

- 검색 API latency, error rate, result count를 기록한다.
- 지도 마커 API 호출 수와 응답 크기를 기록한다.
- 원본 링크 클릭 수를 이벤트/카테고리/지역별로 집계할 수 있어야 한다.
- 수집 작업 성공/실패 수와 실패 원인을 기록한다.

### 보안과 개인정보

- 클릭 로그는 개인 식별 정보를 최소 수집한다.
- IP는 원문 저장 대신 지역 추정 후 폐기하거나 해시/마스킹 정책을 적용한다.
- 관리자 API는 인증과 권한 검사를 필수로 한다.
- 원본 URL은 open redirect 취약점이 생기지 않도록 서버 저장 링크만 허용한다.

### 데이터 품질

- 좌표 누락 이벤트는 지도 검색에 노출하지 않는다.
- 종료된 이벤트는 배치로 자동 상태 전환한다.
- 원본 링크가 일정 기간 접근 불가이면 운영 검수 대상으로 표시한다.
- 중복 이벤트 후보는 운영자가 병합하거나 숨김 처리할 수 있어야 한다.

## 16. 프론트엔드 연동 요구사항

서버는 다음 화면을 지원해야 한다.

- 지도 + 리스트 통합 탐색 화면
- 카테고리/날짜/가격/지역 필터
- 이벤트 상세 화면
- 장소 상세 화면
- 원본 페이지 이동 버튼
- 저장/찜 상태 표시
- 관리자 등록/수정 화면

프론트엔드가 지도 이동 중 API를 자주 호출할 수 있으므로, 서버는 debounce 이후에도
부담이 적은 marker 전용 응답을 제공해야 한다.

## 19. 구현 메모

- 현재 레포에는 `com.lomeone.moyemap` 패키지의 `Venue`, `Review` 초안이 존재한다.
- 실제 제품 목적은 장소 자체보다 이벤트 탐색이므로 `Event`를 핵심 aggregate로 추가해야 한다.
- 기존 `VenueCategory`는 이벤트 카테고리와 장소 카테고리를 분리할지 검토해야 한다.
- `reviews` 테이블명은 다른 제품과 충돌 가능성이 있으므로 `moyemap_reviews`처럼 제품 prefix를 권장한다.
- GraphQL schema도 `Venue` 같은 일반 이름보다 `MoyemapVenue`, `MoyemapEvent`처럼 제품 prefix를 붙이면 충돌 가능성이 낮다.
- MVP에서는 MySQL 범위 검색으로 시작하되, 지도 검색 트래픽이 증가하면 지리 검색 전용 인프라를 도입한다.
